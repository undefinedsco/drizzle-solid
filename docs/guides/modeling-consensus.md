# Modeling Consensus Guide

建模问题不是普通 API 问题。

当问题落在“这个概念该不该挂这个 predicate”“应该复用哪个 IRI”“要不要发明新词汇”时，**单个 AI 的回答只能算草案，不能直接当作仓库权威结论**。

这份文档定义 `drizzle-solid` 当前如何处理这类问题。

## 什么时候进入共识流程

满足以下任一条件，就不要把单次回答直接写进 README、examples 或 public skills：

- 不确定某个 predicate/IRI 是否真的适合该概念
- 不确定某个属性应归属哪个概念
- 标准词汇和项目自定义词汇之间存在冲突
- 不确定是复用现有词汇，还是新建术语
- 现有 docs / examples / skills 对同一建模问题给出了不一致暗示

这类问题应作为：

- `kind:decision`
- `area:modeling`

进入仓库的 decision 处理流程。

## 核心原则

## 当前已稳定的建模结论

下面这条规则已经可以作为仓库权威口径，不再视为未决争议：

- 用 `rdfs:subClassOf` 表达 vocabulary / schema 层级
- 实例默认只持久化一个主、最具体、最权威的 `rdf:type`
- 不默认把父类类型额外物化到实例上
- 不建议再并行引入一套仅仅重复类成员身份的字符串 `type/kind` 系统

如果未来要兼容无推理环境，允许把父类类型落盘，也应作为**显式兼容选项**，而不是默认建模原则

### 1. 单 AI 只负责提案，不负责定案

单个 AI 可以：

- 帮忙澄清问题
- 给出候选 predicate / IRI
- 对比已有 vocabulary
- 写出初步建模建议

但如果问题本身仍有多种合理解释，就不应把这份回答当成最终规范。

### 2. 建模决策要把“描述”与“裁决”分开

当前推荐把角色拆成三层：

- **描述者**：只描述业务场景、约束、反例、期望查询方式
- **提案者**：独立提出候选 IRI / predicate / ownership 方案
- **裁决者**：比较提案，判断是否形成足够稳定的共识

重点是：

- 描述者不要直接指定词汇答案
- 提案者之间要尽量独立
- 裁决阶段要显式处理分歧，而不是默认多数即真理

### 3. 仓库资产只能接收稳定结论

只有当结论已经足够稳定，才应该进入：

- `README.md`
- `docs/guides/`
- `examples/`
- `skills/`

如果还没有共识：

- 可以保留候选方案
- 可以记录取舍理由
- 但不要把它写成唯一推荐答案

## 当前仓库流程

`../consensus` 中已经定义了更完整的多 AI 共识研究方向。当前 `drizzle-solid` 仓库先采用它的治理思想，但暂不假装已经有完整自动化系统。

也就是说，**这里先做流程收口，不做虚假自动化承诺**。

当前实践流程：

1. 用户或维护者提出建模问题
2. 维护者在 triage 时归类为 `kind:decision` + `area:modeling`
3. 收集业务场景（competency question）、约束、候选 IRI 与反例
4. 使用多个独立 AI / 视角做提案与评审
5. 得出以下三种结果之一：
   - 推荐复用某个现有词汇
   - 暂无共识，保留多个候选
   - 现有词汇不足，需要新术语
6. 再把稳定结论回写到 docs / examples / skills / issue 记录
7. 如需长期保留，使用 `docs/guides/decisions/TEMPLATE.md` 沉淀为 decision record

## 对 issue 的要求

建模 issue 不要求提问者先会打 label。

提问者只需要尽量提供：

- 业务场景是什么
- 这个概念要表达什么事实
- 希望支持哪些读写 / 查询方式
- 为什么现有词汇或现有文档不够
- 已考虑过哪些候选词汇

`label` 由维护者在 triage 时补：

- `kind:decision`
- `area:modeling`

## 输出格式建议

对于进入共识流程的问题，最终结论至少应包含：

- 推荐表达的概念定义
- 候选 IRI / predicate 列表
- 为什么采纳当前方案
- 为什么不采纳其他候选
- 这条规则会落到哪些 docs / examples / skills
- 如果仍无共识，当前禁止把什么写成权威口径

## 与 public skills 的关系

public skills 应该消费已经稳定的建模规则，而不是替代共识流程本身。

所以：

- skill 可以帮助提出候选方案
- skill 可以提醒“这个问题需要升级为 decision”
- skill 不应该在建模争议尚未收敛时输出伪确定性的唯一答案
